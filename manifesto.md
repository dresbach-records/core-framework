# Manifesto do CORE

**Server-First Web Framework**

## Introdução

A Web evoluiu.
Mas, em muitos aspectos, ela também se perdeu.

Frameworks modernos resolveram problemas reais, mas ao custo de:

- complexidade excessiva
- estados duplicados
- dependência pesada de JavaScript
- dificuldades de manutenção
- aplicações frágeis a longo prazo

O CORE nasce como uma resposta consciente a esse cenário.

Não como reação emocional.
Não como moda.
**Mas como engenharia deliberada.**

## O que é o CORE

O CORE não é um framework de frontend.
O CORE não é uma SPA.
O CORE não é uma tentativa de reinventar a Web.

O CORE é o **núcleo** de uma aplicação web.

Ele parte de um princípio simples e poderoso:

> O servidor é a fonte da verdade.
> A interface é apenas a projeção desse estado.

## Princípio 1 — Server-First

No CORE, o servidor vem primeiro.

Toda decisão importante acontece no backend:

- estado
- regras
- fluxo
- validação
- autorização

O cliente não governa o sistema.
Ele apenas reflete o que o servidor determina.

## Princípio 2 — HTML como Contrato

HTML não é um detalhe.
HTML é o contrato entre o sistema e o usuário.

No CORE:

- o HTML é semântico
- o HTML é completo
- o HTML é entregue pronto
- o HTML não depende de hidratação

JavaScript não é pré-requisito para que o conteúdo exista.

## Princípio 3 — Estado Único

Estados duplicados são a origem de bugs complexos.

No CORE:

- existe um único estado
- esse estado vive no servidor
- ele é previsível
- ele é auditável
- ele é controlável

Não há sincronização frágil entre cliente e servidor, porque não há duplicação.

## Princípio 4 — JavaScript é Opcional

JavaScript não é proibido.
Mas também não é obrigatório.

No CORE:

- JavaScript é utilitário
- nunca estrutural
- nunca a fonte da verdade
- nunca o controlador da aplicação

Se JavaScript falhar, a aplicação continua funcional.

## Princípio 5 — Previsibilidade acima de efeitos

Interfaces podem ser bonitas.
Mas sistemas precisam ser previsíveis.

O CORE prioriza:

- clareza
- comportamento consistente
- respostas imediatas
- menos “mágica”
- mais controle

Uma aplicação previsível é mais fácil de manter, escalar e evoluir.

## Princípio 6 — Acessibilidade e Padrões Web

A Web é aberta porque segue padrões.

O CORE respeita:

- HTML semântico
- HTTP correto
- CSS padrão
- acessibilidade nativa
- indexação por mecanismos de busca

O CORE não exige hacks para funcionar bem na Web.
Ele usa a Web como ela foi projetada.

## Princípio 7 — Longevidade como Meta

O CORE não é feito para impressionar em demos.

Ele é feito para:

- rodar por anos
- sobreviver a mudanças de equipe
- facilitar manutenção
- reduzir dívida técnica
- envelhecer bem

Longevidade não é um efeito colateral.
**É um objetivo explícito.**

## O que o CORE não pretende ser

O CORE não pretende:

- substituir todos os frameworks
- competir por hype
- vencer debates de rede social
- ser a escolha universal

O CORE existe para quem:

- constrói produtos reais
- mantém sistemas vivos
- prioriza estabilidade
- prefere clareza a complexidade

## Origem

O CORE foi desenvolvido a partir de necessidades reais, em ambientes de produção, dentro da **Dresbach Hosting do Brasil**.

Ele nasce da prática, não da teoria.

## Compromisso

O CORE assume um compromisso com:

- simplicidade consciente
- engenharia responsável
- respeito aos padrões
- clareza arquitetural
- evolução cuidadosa

Cada decisão deve tornar o sistema mais compreensível, não mais impressionante.

## Declaração Final

O CORE não tenta controlar o navegador.
Ele organiza o servidor.

O CORE não empurra lógica para o cliente.
Ele assume responsabilidade.

O CORE não segue tendências.
Ele constrói fundamentos.

Este é o CORE.
**O núcleo antes do excesso.**
**A base antes do brilho.**
**A engenharia antes do hype.**
